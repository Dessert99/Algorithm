[https://leetcode.com/problems/human-traffic-of-stadium/description/](https://leetcode.com/problems/human-traffic-of-stadium/description/)
```sql
WITH
    -- 1. id - ROW_NUMBER() OVER() 문법
    add_group_key AS (
        SELECT
            id,
            visit_date,
            people,
            -- 연속된 id라면, group_key는 동일한 숫자이다.
           id - ROW_NUMBER() OVER (ORDER BY id) AS group_key
        FROM
            Stadium
        WHERE
            people >= 100 -- 여기서 인구 수 필터링
    ),
    -- 2. COUNT() OVER()로 개수 컬럼 추가
    count_peeople AS (
        SELECT
            id,
            visit_date,
            people,
            COUNT(*) OVER(PARTITION BY group_key) AS count_id
        FROM
            add_group_key
    )

SELECT
    id, 
    visit_date,
    people
FROM
    count_peeople
WHERE
    count_id >= 3
ORDER BY
    visit_date
```
### 🔗 풀이
1. 🤔 연속된 ID행을 어떻게 판단하는가?
    - LEAD를 사용해야 하는가? -> ❌
    - 횟수를 알아야 하기 때문에 COUNT() OVER()을 사용해야 하는가? -> ✅

2. 🤔 COUNT() OVER() 윈도우 함수로 새로운 컬럼을 만들어보자?
     - 중요한 사실이 하나 더 있다. ROW_NUMBER() 함수도 사용해야 한다. 무엇을 그룹해서 셀 것인지 판단해야 하기 떄문이다.

### 🔗 배운점
1. `ROW_NUMBER()`
    ```sql
        ROW_NUMBER() OVER (
            PARTITION BY 그룹핑할_컬럼    -- (선택) 이 컬럼 기준으로 그룹을 나누고, 그룹마다 1번부터 리셋
            ORDER BY 정렬할_컬럼 ASC|DESC -- (필수) 어떤 순서로 1, 2, 3... 번호를 매길지 결정
        )
    ```
    - `ROW_NUMBER()`: 번호를 매기겠다는 함수 자체입니다. 괄호 () 안에는 아무것도 넣지 않습니다.
    - `OVER (...)`: 윈도우 함수의 핵심입니다. "어떤 기준을 바탕으로(OVER) 번호를 매길까?"를 정의하는 공간입니다.
    - `PARTITION BY` (선택 사항): 특정 그룹별로 번호를 따로따로 매기고 싶을 때 씁니다.
        - 예를 들어 반별로 1등, 2등을 따로 매기고 싶다면 `PARTITION BY` 반을 씁니다. 반이 바뀌면 번호는 다시 1번부터 시작합니다.
        - 생략하면 전체 데이터를 통째로 하나의 그룹으로 봅니다. (우리가 푼 문제에서는 생략했습니다.)
    - `ORDER BY` (필수 사항): 1, 2, 3번을 매기는 '순서 기준'입니다.
        - 이걸 생략하면 데이터베이스 마음대로 뒤죽박죽 번호를 매기려고 하기 때문에 대부분의 SQL에서는 에러를 냅니다. 무조건 적어줘야 합니다.
2. 테이블 예시
   - `id-ROW_NUMBER()`
     ```sql
        | **상태** | **id (보폭 +1)** | **ROW_NUMBER() (보폭 +1)** | **두 숫자의 차이 (id - ROW_NUMBER)** |
        | --- | --- | --- | --- |
        | **연속** | 100 | 1 | **99** |
        | **연속** | 101 | 2 | **99** |
        | **연속** | 102 | 3 | **99** |
     ```
   - 중간에 누군가 훅 건너뛸 때
     ```sql
        | **상태** | **id** | **ROW_NUMBER()** | **두 숫자의 차이 (id - ROW_NUMBER)** |
        | --- | --- | --- | --- |
        | 연속 | 100 | 1 | **99** (A그룹) |
        | 연속 | 101 | 2 | **99** (A그룹) |
        | 연속 | 102 | 3 | **99** (A그룹) |
        | **단절!** | **105** *(+3 증가)* | **4** *(+1 증가)* | **101** (새로운 B그룹 탄생!) |
        | 연속 | 106 | 5 | **101** (B그룹) |
        | 연속 | 107 | 6 | **101** (B그룹) |
     ```
       - `ROW_NUMBER()`는 앞 상황이 어찌 됐든 눈치 없이 무조건 1만 증가해서 4가 된다.
       - `id`는 보폭을 3이나 뛰었는데, `ROW_NUMBER`는 보폭을 1만 뛰었으니 둘 사이의 거리(차이)가 99에서 101로 틀어져 버립니다. 하지만 105번 이후로 106, 107이 다시 1씩 예쁘게 연속으로 증가한다면? 그 구간 안에서는 다시 둘의 보폭이 같아지니 차이가 101로 고정
